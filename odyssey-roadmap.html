Technology Stack

Philosophy: One language end-to-end. Zero infrastructure complexity until we need it. Every choice optimizes for shipping speed and near-zero monthly cost during development.
Frontend
React + Vite
TypeScript throughout. Vite's dev server is instant. Same codebase wraps into desktop via Tauri in Phase 3 — no rewrite needed.
Fast Dev
Styling
Tailwind CSS
Utility-first means we build custom UI fast without fighting a component library. We control every pixel — important for Odyssey's design identity.
Ship Fast
Backend API
Node + Fastify
TypeScript backend means one language everywhere. Fastify is 2x faster than Express with better schema validation. Handles webhooks from GitHub and Microsoft.
Core
Database + Auth
Supabase
PostgreSQL + Auth + Realtime + file storage in one free tier. Handles GitHub OAuth and Microsoft OAuth out of the box. Scales to paid without migration.

AI Layer — Claude Integration

Token Budget Per Query

What gets sent to Claude for a typical "project status" request:

System prompt + project context~500 tokens
Last 20 GitHub commits~800 tokens
Recent Teams / doc events~600 tokens
Goals + deadlines JSON~400 tokens
User's question~30 tokens
Claude's response (output)~400 tokens
Total per query~2,730 tokens
Cost Per User Per Month

At claude-sonnet-4-5 pricing ($3/M input, $15/M output):

Cost per query (input)~$0.007
Cost per query (output)~$0.006
Cost per query (total)~$0.013
3 queries/day × 30 days~$1.17/mo
5 queries/day × 30 days~$1.95/mo
Break-even at $15/mo subscription> 750 queries
Margin at avg usage> 85%
AI Query Types — Phase 1
  • → Activity Summary — "What happened on this project this week?"
  • → Deadline Risk — "Are we on track to hit our March 30 deadline?"
  • → Contribution Attribution — "Who drove the most progress on the backend?"
  • → Weekly Digest — Email narrative of the project's progress
  • → Project History — "Tell the story of how this project came together"
Prompt Architecture Strategy

Claude gets a structured JSON payload — never raw log data. We pre-process events into a normalized summary before sending. This keeps tokens low and responses sharp.

Each query type has its own system prompt template. Project-level context is cached — we don't re-send the full project description on every query, only the delta since last call.

Key rule: Never send more than 4,000 input tokens per query during MVP. If the project is large, we summarize older events before including them. This caps worst-case cost at ~$0.10 per query.

Core Database Schema

The most important decision in this entire codebase. A unified events table is what makes Odyssey work. Every source — GitHub, Teams, OneDrive, OneNote — normalizes into this one structure. The AI layer reads only from here.
-- The unified event log. Every integration writes here. CREATE TABLE events ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), project_id UUID REFERENCES projects(id) ON DELETE CASCADE, actor_id UUID REFERENCES users(id), source TEXT NOT NULL, -- 'github' | 'teams' | 'onedrive' | 'onenote' | 'manual' event_type TEXT NOT NULL, -- 'commit' | 'message' | 'file_edit' | 'note' | 'meeting' title TEXT, summary TEXT, -- pre-processed for AI context metadata JSONB, -- source-specific raw data occurred_at TIMESTAMPTZ NOT NULL, created_at TIMESTAMPTZ DEFAULT now() ); -- Projects CREATE TABLE projects ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), name TEXT NOT NULL, description TEXT, owner_id UUID REFERENCES users(id), created_at TIMESTAMPTZ DEFAULT now() ); -- Goals with deadline tracking CREATE TABLE goals ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), project_id UUID REFERENCES projects(id), title TEXT NOT NULL, deadline DATE, status TEXT DEFAULT 'active', -- 'active' | 'at_risk' | 'complete' | 'missed' risk_score FLOAT, -- 0.0–1.0, updated by AI analysis progress INT DEFAULT 0 -- 0–100 ); -- Connected integrations per project CREATE TABLE integrations ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), project_id UUID REFERENCES projects(id), type TEXT NOT NULL, -- 'github' | 'teams' | 'onedrive' config JSONB, -- repo name, channel ID, folder path etc. token_ref TEXT -- reference to encrypted token in vault );

Build Principles

01
Unified Event Schema First
Every integration — GitHub, Teams, OneDrive, OneNote — writes to one events table with a consistent shape. Don't let this slip. It's the foundation everything else is built on.
02
Ship Real Data Fast
Mock data is fine for UI weeks 1–2. But by Week 4, Claude must be looking at real GitHub commits. Fake data creates fake confidence. Real data reveals real problems early.
03
Cap AI Token Cost
Every Claude query gets a 4,000 input token budget. Pre-process and summarize before sending. Never stream raw event logs to the API. This keeps costs predictable at any scale.
04
Desktop is a Wrapper, Not a Rewrite
Tauri wraps the React frontend. No native-specific code in the UI layer. When we add the desktop build in Week 11, the frontend ships unchanged. This is why we choose Tauri over Electron alternatives.
05
Never Store OAuth Tokens in DB
GitHub and Microsoft access tokens go in Supabase Vault (encrypted secrets), never in the integrations table directly. token_ref holds a key name, not the token itself. Non-negotiable for security.
06
One Language Everywhere
TypeScript from the React component to the Fastify route to the Supabase type. No context switching. No Python microservices until we genuinely need heavy data science. Shipping speed over architectural elegance.
